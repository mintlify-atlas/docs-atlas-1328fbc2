The Flutter Billing App is built using **Clean Architecture** principles with a **Feature-First** organizational structure. This ensures scalability, maintainability, and clear separation of concerns across all modules.

## Architectural Overview

The application follows a layered architecture pattern where each layer has specific responsibilities and dependencies flow inward toward the domain layer.

### Key Architectural Choices

<AccordionGroup>
  <Accordion title="State Management: flutter_bloc">
    Uses the BLoC (Business Logic Component) pattern for predictable state management. All business logic is isolated in BLoCs, making the codebase testable and maintainable.
    
    - Immutable states using `equatable`
    - Event-driven architecture
    - Clear separation from UI
  </Accordion>
  
  <Accordion title="Dependency Injection: get_it">
    Service locator pattern for managing dependencies throughout the app. All repositories, use cases, and BLoCs are registered in a central service locator.
    
    ```dart
    final sl = GetIt.instance;
    
    sl.registerFactory(() => ProductBloc(
      getProductsUseCase: sl(),
      addProductUseCase: sl(),
    ));
    ```
  </Accordion>
  
  <Accordion title="Functional Error Handling: fpdart">
    Uses `Either<Failure, Success>` pattern instead of throwing exceptions. All use cases return `Either` types for explicit error handling.
    
    ```dart
    Future<Either<Failure, List<Product>>> getProducts();
    ```
  </Accordion>
  
  <Accordion title="Offline-First: Hive">
    Lightning-fast NoSQL database that stores data locally. No network dependency required for core operations.
    
    - Type-safe with code generation
    - Fast read/write operations
    - Minimal setup overhead
  </Accordion>
</AccordionGroup>

## Clean Architecture Layers

### Domain Layer (Business Logic)

The domain layer contains the core business logic and is completely independent of external frameworks.

#### Entities

Pure Dart classes representing business objects:

```dart lib/features/product/domain/entities/product.dart
class Product extends Equatable {
  final String id;
  final String name;
  final String barcode;
  final double price;
  final int stock;

  const Product({
    required this.id,
    required this.name,
    required this.barcode,
    required this.price,
    this.stock = 0,
  });

  @override
  List<Object?> get props => [id, name, barcode, price, stock];
}
```

#### Use Cases

Encapsulate single business operations. Each use case implements the `UseCase` interface:

```dart lib/core/usecase/usecase.dart
abstract class UseCase<Type, Params> {
  Future<Either<Failure, Type>> call(Params params);
}

class NoParams extends Equatable {
  @override
  List<Object> get props => [];
}
```

Example use case implementation:

```dart lib/features/product/domain/usecases/product_usecases.dart
class GetProductsUseCase implements UseCase<List<Product>, NoParams> {
  final ProductRepository repository;

  GetProductsUseCase(this.repository);

  @override
  Future<Either<Failure, List<Product>>> call(NoParams params) {
    return repository.getProducts();
  }
}
```

#### Repository Interfaces

Abstract contracts defining data operations:

```dart lib/features/product/domain/repositories/product_repository.dart
abstract class ProductRepository {
  Future<Either<Failure, List<Product>>> getProducts();
  Future<Either<Failure, void>> addProduct(Product product);
  Future<Either<Failure, void>> updateProduct(Product product);
  Future<Either<Failure, void>> deleteProduct(String id);
  Future<Either<Failure, Product>> getProductByBarcode(String barcode);
}
```

#### Repository Implementation

Concrete implementations using Hive for data persistence:

```dart lib/features/product/data/repositories/product_repository_impl.dart
class ProductRepositoryImpl implements ProductRepository {
  @override
  Future<Either<Failure, List<Product>>> getProducts() async {
    try {
      final box = HiveDatabase.productBox;
      final products = box.values.toList();
      return Right(products);
    } catch (e) {
      return Left(DatabaseFailure('Failed to load products'));
    }
  }

  @override
  Future<Either<Failure, void>> addProduct(Product product) async {
    try {
      final box = HiveDatabase.productBox;
      final model = ProductModel(
        id: product.id,
        name: product.name,
        barcode: product.barcode,
        price: product.price,
        stock: product.stock,
      );
      await box.put(product.id, model);
      return const Right(null);
    } catch (e) {
      return Left(DatabaseFailure('Failed to add product'));
    }
  }
}
```

### Presentation Layer (UI & State Management)

The presentation layer handles user interface and state management using BLoC.

#### BLoC Pattern

Each feature has a dedicated BLoC for managing state:

```dart lib/features/product/presentation/bloc/product_bloc.dart
class ProductBloc extends Bloc<ProductEvent, ProductState> {
  final GetProductsUseCase getProductsUseCase;
  final AddProductUseCase addProductUseCase;
  final UpdateProductUseCase updateProductUseCase;
  final DeleteProductUseCase deleteProductUseCase;

  ProductBloc({
    required this.getProductsUseCase,
    required this.addProductUseCase,
    required this.updateProductUseCase,
    required this.deleteProductUseCase,
  }) : super(const ProductState()) {
    on<LoadProducts>(_onLoadProducts);
    on<AddProduct>(_onAddProduct);
    on<UpdateProduct>(_onUpdateProduct);
    on<DeleteProduct>(_onDeleteProduct);
  }

  Future<void> _onLoadProducts(
      LoadProducts event, Emitter<ProductState> emit) async {
    emit(state.copyWith(status: ProductStatus.loading));
    final result = await getProductsUseCase(NoParams());
    result.fold(
      (failure) => emit(state.copyWith(
        status: ProductStatus.error, 
        message: failure.message
      )),
      (products) => emit(state.copyWith(
        status: ProductStatus.loaded, 
        products: products
      )),
    );
  }
}
```

#### State Objects

Immutable state classes using Equatable:

```dart
enum ProductStatus { initial, loading, loaded, success, error }

class ProductState extends Equatable {
  final ProductStatus status;
  final List<Product> products;
  final String? message;

  const ProductState({
    this.status = ProductStatus.initial,
    this.products = const [],
    this.message,
  });

  ProductState copyWith({
    ProductStatus? status,
    List<Product>? products,
    String? message,
  }) {
    return ProductState(
      status: status ?? this.status,
      products: products ?? this.products,
      message: message ?? this.message,
    );
  }

  @override
  List<Object?> get props => [status, products, message];
}
```

## Dependency Injection

All dependencies are registered in the service locator at app startup:

```dart lib/core/service_locator.dart
import 'package:get_it/get_it.dart';

final sl = GetIt.instance;

Future<void> init() async {
  // BLoCs - Registered as factories (new instance per request)
  sl.registerFactory(
    () => ProductBloc(
      getProductsUseCase: sl(),
      addProductUseCase: sl(),
      updateProductUseCase: sl(),
      deleteProductUseCase: sl(),
    ),
  );

  sl.registerFactory(
    () => ShopBloc(
      getShopUseCase: sl(),
      updateShopUseCase: sl(),
    ),
  );

  // Use cases - Registered as lazy singletons
  sl.registerLazySingleton(() => GetProductsUseCase(sl()));
  sl.registerLazySingleton(() => AddProductUseCase(sl()));
  sl.registerLazySingleton(() => UpdateProductUseCase(sl()));
  sl.registerLazySingleton(() => DeleteProductUseCase(sl()));
  sl.registerLazySingleton(() => GetProductByBarcodeUseCase(sl()));

  // Repositories - Registered as lazy singletons
  sl.registerLazySingleton<ProductRepository>(
    () => ProductRepositoryImpl(),
  );

  sl.registerLazySingleton<ShopRepository>(
    () => ShopRepositoryImpl(),
  );
}
```

<Warning>
BLoCs are registered as factories to ensure each widget gets a fresh instance, while use cases and repositories are singletons to share state across the app.
</Warning>

## Error Handling Strategy

The app uses functional error handling with `fpdart`'s `Either` type:

```dart lib/core/error/failure.dart
abstract class Failure extends Equatable {
  final String message;
  
  const Failure(this.message);
  
  @override
  List<Object> get props => [message];
}

class DatabaseFailure extends Failure {
  const DatabaseFailure(String message) : super(message);
}

class NotFoundFailure extends Failure {
  const NotFoundFailure(String message) : super(message);
}
```

### Handling Results in BLoC

Use cases return `Either<Failure, Success>` which BLoCs handle explicitly:

```dart
final result = await getProductsUseCase(NoParams());
result.fold(
  (failure) => emit(state.copyWith(
    status: ProductStatus.error,
    message: failure.message,
  )),
  (products) => emit(state.copyWith(
    status: ProductStatus.loaded,
    products: products,
  )),
);
```

<Note>
This approach makes error handling explicit and prevents uncaught exceptions from crashing the app.
</Note>

## Routing Architecture

The app uses `go_router` for declarative, type-safe navigation:

```dart lib/config/routes/app_routes.dart
final router = GoRouter(
  initialLocation: '/',
  routes: [
    GoRoute(
      path: '/',
      builder: (context, state) => const HomePage(),
      routes: [
        GoRoute(
          path: 'scanner',
          builder: (context, state) => const ScannerPage(),
        ),
        GoRoute(
          path: 'checkout',
          builder: (context, state) => const CheckoutPage(),
        ),
      ],
    ),
    GoRoute(
      path: '/products',
      builder: (context, state) => const ProductListPage(),
      routes: [
        GoRoute(
          path: 'add',
          builder: (context, state) => const AddProductPage(),
        ),
        GoRoute(
          path: 'edit/:id',
          builder: (context, state) {
            final product = state.extra as Product?;
            if (product == null) {
              return const ProductListPage();
            }
            return EditProductPage(product: product);
          },
        ),
      ],
    ),
  ],
);
```

### Navigation Patterns

- **Nested Routes**: Child routes like `/scanner` under `/`
- **Route Parameters**: Dynamic segments like `:id` in edit routes
- **Extra Data**: Pass objects via `state.extra` for complex data
- **Deep Linking**: Full URL support for direct navigation

## Performance Optimizations

### Hive Database
- **Fast reads**: In-memory caching for frequently accessed data
- **Lazy loading**: Boxes opened on-demand
- **Compact storage**: Binary serialization reduces storage size

### BLoC Optimization
- **Factory pattern**: New instances prevent state leaks between screens
- **Equatable**: Prevents unnecessary rebuilds by comparing state values
- **Selective emissions**: Only emit when state actually changes

### UI Performance
- **Const constructors**: Reduce widget rebuilds
- **ListView builders**: Lazy rendering for large product lists
- **Cached fonts**: Google Fonts cached locally after first load
